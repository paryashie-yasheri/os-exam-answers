## 1. Архитектура компьютерных систем. Архитектура Фон-Неймана и Гарвардская архитектура. Принципы архитектуры Фон-Неймана. Архитектуры NUMA и UMA.
![[Harvard-VonNeumann.png]]
### Два основных вида архитектур:
#### Гарвардская
Принципы
- Данные и команды хранятся в разной памяти
- В отличии от Фон-Неймана, каналы для передачи данных из inctruction memory и data memory разные
#### Фон-Неймана
Принципы:
- Однородность памяти
	- Команды и данные хранятся в одной и той же памяти, внешне не различимы
- Адресности
	- Память состоит из пронумерованных ячеек, процессору доступна любая ячейка
- Программного управления
	- Вычисления представлены в виде программы, состоящей из последовательности команд
- Двоичного кодирования
	- То есть вся информация (и  данные и команды кодируются двоичным кодом (1 и 0))

### Разделение архитектур по организации памяти (UMA/NUMA)
#### UMA
![[UMA.png]]
- Uniform Memory Access / Однородный Доступ к Памяти
- Все устройства одного ранга
- Все устройства имеют одинаковый доступ к памяти
- Время доступа и задержка до памяти одинаковое
- +:
	1. Простота реализации
	2. Одинаковое время доступа к памяти
 - -: 
	 1. Плохая масштабируемость
	 2. Нормально работает только при <=4 процессоров

#### NUMA
![[NUMA.png]]
- Все системные ресурсы расположены на отдельных системных платах
- Имеется локальная память 
	- Находится ближе к CPU
- Общая память, устройства I/O доступны через коммутатор (Crossbar)
	- Коммутатор координирует вазаимодействие компонентов
- Используется в серверах
- **NB:** Адресное пространство общее для всех процессов
- +:
	1. Возможность хотсвапа
	2. Хорошая масштабируемость
	3. Надежность
	4. Отказоустойчивость
- -:
	1. Требовательность к ОС
		- ОС должна поддерживать все фишки
	2. Необходимость учитывания времени доступа и расположения данных в памяти
		 - Т.к. время доступа зависит от того, где данные расположены

## 3. Организация прерываний, типы прерываний, контроллер прерываний.
![[interrupts.png]]
[Прерывание](https://ru.wikipedia.org/wiki/%D0%9F%D1%80%D0%B5%D1%80%D1%8B%D0%B2%D0%B0%D0%BD%D0%B8%D0%B5) -- одна из базовых концепций ВТ. Суть: при наступлении какого-либо события происходит передача управления специальной процедуре, называемой обработчиком прерывания (ISR, Interrupt Service Routine). В отличии от условных и безусловных переходов, прерывание может быть вызывано в любом месте программы. Как правило, после обработки прерывания, ISR возвращает управление прерванной программе
Прерывание извещает процессор о наступлении высокоприоритетного события, требующего прерывания текущего кода, выполняемого процессором. Процессор отвечает приостановкой текущей активности, сохраняя свое состояние и выполняя функцию, называемую обработчиком прерывания, которая реагирует на событие и обслуживает его, после чего возвращает управление в прерванный код

Прерывания:
- Могут быть вызваны самой программой или сигналом с устройств IO
- Более высокий приоритет прерываний могут прерывать более низкие
- Выполняются в конце цикла команды
- Могут быть вложенными
	- Существуют ограничения по вложенности обработчика в зависимости от архитектуры

## 4. Типичные функции операционной системы. Интерфейсы ОС. Работа ОС как замена оператора ЭВМ.
### Типичные функции ОС
- Разработка программ
- Выполнение программ
- Доступ к устройствам IO
- Контролируемый доступ к файлам
- Доступ к системе и системным ресурсам
- Обнаружение и обработка ошибок 
- Учет использования и диспетчеризация ресурсов
### Интерфейсы ОС
![[API_ABI_ISA.png]]
#### ISA
- Instruction Set Architecture
- Набор команд
- Определяет набор команд машинного языка, которые может выполнять компьютер
- Этот интерфейс является границей между аппаратным и программным обеспечением
- На этом уровне определяются реализованные в микропроцессоре конкретного типа%
	- архитектура памяти
	- взаимодействие с внешними устройствами IO
	- режимы адресации
	- регистры
	- машинные команды 
	- различные типы внутренних данных (e.g. float, целочисленные, etc)
#### ABI
- Application Binary interface
- Бинарный интерфейс приложений
- Набор соглашений для доступа приложения к ОС и другим низкоуровневым сервисам, спроектированный для переносимости исполняемого кода между машинами, имеющими совместимые ABI
- Разница с API:
- В отличие от API, который регламентирует совместимость на уровне исходного текста, ABI можно рассматривать как набор правил, позволяющих компоновщику объединять откомпилированные модули компонента без перекомпиляции всего иходного текста, в то же время определяя двоичный интерфейс
#### API
- Application Programming Interface
- Интерфейс прикладных программ
- Проще говоря, это способ взаимодействия какого-то программного кода с набором каких-то программных компонентов, с помощью которых одна компьютерная программа (например, бот или сайт) может взаимодействовать с другой программой.
### Работа ОС как замена оператора ЭВМ
В первых ЭВМ был только пульт управления
Обязанности операторов были:
- Получение программы с данными от программиста
- Подготовка программы к загрузке (e.g. перфокарт)
- Загрузка программы и компилятора
- Запуск программы на вычисление
- Передача распечатки результатов программисту
Минусы операторов:
- Наличие расписания машинного времени
- Долгое время подготовки к работе

1) Монолитное ядро
	- Большое ядро, содержащее практически всю ОС, включая планирование, файловую систему, драйверы устройств, управление памятью. 
	- Все функциональные компоненты ядра имеют доступ ко всем его внутренний структурам данных и процедурам
	- Как правило, монолитное ядро реализовано как единый процесс со всеми элементами, разделяющими одно и то же адресное пространство
2) Модульное ядро
	- Современная, усовершенствованная модификация архитектуры монолитных ядер ОС
	- В отличии от монолитов, модульные, как правило не требуют полной перекомпиляции ядра при изменении состава аппаратного обеспечения компьютера. Вместо этого может хотсвапать модули
	- +:
		- нет необходимости собирать ядро при установке нового аппаратного устройства
	- -:
		- Скорость работы ядра
3) Микроядро
	- Предоставляет только элементарные функции управления процессами и минимальный набор абстракций для работы с оборудованием. 
	- Бо́льшая часть работы осуществляется с помощью специальных пользовательских процессов, называемых сервисами. 
	- Решающим критерием «микроядерности» является размещение всех или почти всех драйверов и модулей в сервисных процессах, иногда с явной невозможностью загрузки любых модулей расширения в собственно микроядро, а также разработки таких расширений.
4) Гибридное ядро
	- Модифицированные микроядра, позволяющие для ускорения работы запускать «несущественные» части в пространстве ядра.
5) Есть еще экзоядра и наноядра, ксв про них не рассказывал на лекциях + этого нет у столингса, но на экзе имхо можно упомянуть

 Отличительной особенностью большинства операционных систем до сегодняшнего дня является большое монолитное ядро. Ядро операционной системы обеспечивает большинство ее возможностей, включая планирование, работу с файловой системой, сетевые функции, работу драйверов различных устройств, управление памятью и многие другие. Обычно монолитное ядро реализуется как единый процесс, все элементы которого используют одно и то же адресное пространство. В архитектуре микроядра ядру отводится лишь несколько самых важных функций, в число которых входят управление адресным пространством, обеспечение взаимодействия между процессами (interprocess communication - IPC) и основное планирование. Работу других сервисов операционной системы обеспечивают процессы, которые иногда называют серверами. Эти процессы запускаются в пользовательском режиме, и микроядро работает с ними так же, как и с другими приложениями. Такой подход позволяет разделить задачу разработки операционной системы на разработку ядра и разработку сервера. Серверы можно настраивать для требований конкретных приложений или среды. Выделение в структуре системы микроядра упрощает реализацию системы, обеспечивает ее гибкость, а также хорошо вписывается в распределенную среду. Фактически микроядро взаимодействует с локальным и удаленным серверами по одной и той же схеме, что упрощает построение распределенных систем.
